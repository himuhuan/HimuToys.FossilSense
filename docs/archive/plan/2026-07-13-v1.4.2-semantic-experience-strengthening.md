# v1.4.2 语义与体验加强计划

> 状态：implemented / 已实现并归档
>
> 适用分支：`release/v1.4.2`
>
> 基线：FossilSense `1.4.1`、SQLite schema `15`、Call Relations protocol `v2`
>
> 计划日期：2026-07-13
>
> 完成日期：2026-07-14
>
> 本文是已经完成的施工计划，仅供追溯；当前能力以 `CLAUDE.md`、`README.md`、`extensions/vscode/README.md`、代码与测试为准。

## 目标

FossilSense 当前已经有函数符号、调用点、arity、include 可达图、record、alias 和 Rich Hover 等事实，但这些事实没有进入同一条请求期候选链。结果是同一个名字在不同功能里会被重复解释：

- Signature Help 能看到多个函数签名，但不知道哪些 `.h/.c` 实际是同一个函数，也没有充分使用调用参数个数。
- Hover 和 Go to Definition 仍主要从 `SymbolRecord` 出发，因此已经存在的 `CallSiteFact.argument_count` 与 `SignatureShape` 没有参与筛选。
- `.h/.c` 配对目前是同 `ProjectKey` 下的文档来源偏好，不检查 include 可达性和一对一关系，因此它更像展示启发式，而不是可解释的声明/定义对应关系。
- `RecordDef` 已经保存 record 源码范围，但 Hover 只读取普通 symbol signature，无法稳定展示保留原排版的完整 struct。
- `type_aliases` 可以递归找到终端 record，但当前 alias fact 没有完整 declarator 形状，因此复杂 typedef 无法准确生成 `(aka. ...)`。

本次工作的目标不是在现有三个 handler 上继续叠加条件，而是建立一个**共享的请求期候选解析层**：

```text
RequestContext
    + generation-pinned store views
    + workspace dirty-document overlay
    + ReachGraph
    + current call/type context
        |
        v
CandidateQueryService
    |-- CallableCandidateSet
    |     |-- arity compatibility
    |     |-- declaration/definition counterpart graph
    |     `-- feature-specific presentation
    |
    `-- TypeCandidateSet
          |-- record resolution
          |-- alias-chain resolution
          `-- source-range hydration
```

一旦这条链建立，Hover、Definition、Signature Help、Completion Documentation 和 Call Hierarchy 不再各自决定“哪个函数算同一个”“哪个签名应该显示”。它们共享候选事实，只保留各自的展示政策。

## 版本定位

v1.4.2 是**语义候选一致性和交互体验版本**。这里的“语义”仍然是 FossilSense 的 best-effort candidate semantics，不是编译器级 binding。

本版本可以根据以下证据收窄或归并候选：

- exact name；
- 当前文件与 include 可达性；
- 调用表达式的参数个数；
- 函数声明的 `min_arity / max_arity / variadic`；
- declaration/definition role；
- header/source 文件角色；
- 规范化完整签名；
- linkage；
- alias chain 与唯一终端 record。

本版本仍然不承诺：

- 根据实参类型执行 C++ overload resolution；
- 模板实例化、ADL、namespace using、继承和访问控制；
- 宏展开后的调用绑定；
- 函数指针、callable object 或任意表达式返回类型推断；
- 把名字候选伪装成唯一编译语义结果。

这意味着 arity 可以排除“明确不可能”的函数，但 `foo(1)` 仍然无法仅凭 `1` 判断应该选择 `foo(int)` 还是 `foo(long)`。

## 全局设计原则

### 候选事实只能有一个主来源

自由函数已经有 `CallableAnchor`、`CallableEntity`、`CallSiteFact` 和 `SignatureShape`。继续让 Hover、Definition 和 Signature Help 以 `SymbolRecord.signature` 为函数主事实，会让 call relation 与普通查询形成两套函数语义。

因此 v1.4.2 直接做以下迁移：

- 函数 Hover、Definition、Signature Help 和函数 completion documentation 以 `CallableAnchor` 为主事实。
- `DefinitionCandidate` 继续作为跳转候选的规范 DTO，由 `CallableAnchor` 投影得到，不新增平行的 `SemanticDefinition`。
- 非函数 symbol 继续使用 `SymbolRecord`。
- `SymbolRecord` 的函数行可以暂时保留用于 NameTable、workspace symbol 和旧测试 oracle，但不能继续作为新函数查询政策的事实来源。
- 迁移完成后删除只服务旧函数查询链的 rank/hydration helper，不保留两套生产路径。

### 筛选、归并和展示必须分层

当前 definition ranker 同时影响 Definition、Hover 和 Signature Help，因此 `.c` definition 的导航加分也泄漏到了 Hover。

新链路必须分为三步：

```text
候选召回
    -> 上下文兼容性筛选
    -> 逻辑实体/对侧锚点归并
    -> consumer presentation policy
```

其中：

- **筛选**回答“这个候选是否已被现有证据排除”。
- **归并**回答“两个锚点是否可以视为同一个逻辑函数的 declaration/definition variants”。
- **展示**回答“当前功能应该显示 header、source，还是全部 variants”。

Definition 可以 source-first，而 Hover 可以 header-first；这不意味着两个功能使用了不同的函数身份。

### 允许破坏性迁移，不维护错误兼容

本次计划不以保持旧内部结构、旧排序偶然行为或旧 SQLite shape 为目标。

- schema 直接升级到 `16`。
- schema mismatch 走全量旁路重建，不实现 schema 15/16 双读和兼容 view。
- completion resolve data 可以升级内部版本；扩展和 server 同包发布，不保留旧 data adapter。
- `.h/.c` 旧的 `ProjectKey`-only pairing 在新 resolver 落地后删除。
- 旧测试如果验证的是不再成立的偶然顺序，应改为验证新行为合同，而不是给旧行为加 feature flag。

这里仍然保留的是**数据发布安全性**，不是旧语义兼容性。旧 generation 的 lease 可以继续读完，但新版本不会把旧 schema 当作可长期服务的格式。

### 唯一性必须能解释

“唯一候选”不能等同于 `Vec` 排序后的第一项。

任何直接跳转、`.h/.c` 折叠或 typedef `aka` 都必须满足明确的 uniqueness contract：

- 候选召回没有因预算截断；
- 相关 scope 没有 unresolved/ambiguous include 或遍历上限导致的开放性；
- 同一最高证据层只有一个逻辑目标；
- 归并前按稳定 id 去重；
- 如果无法证明唯一，则返回候选或降级，不把排序第一项写成确定关系。

### 请求必须绑定同一代事实

候选筛选会同时读取 callable anchor、call site、record、alias、ReachGraph 和当前 buffer。如果这些数据来自不同 generation，最容易出现的结果就是范围错位和错误配对。

因此所有入口继续遵守：

- 请求开始时捕获一个 `RequestContext`。
- SQLite 查询使用同一 `SemanticGeneration`。
- `ReachGraph`、project context 和 call read handle 来自同一 `EngineSnapshot`。
- dirty-document overlay 绑定独立单调 `overlay epoch`。
- 源码 range hydration 必须验证它对应的 revision 或当前 buffer version。

## 当前实现问题

### 函数查询仍分成两套事实

`parser/callables.rs` 已经为函数生成：

- `CallableAnchor.role`；
- `LinkageDomain`；
- `SignatureShape`；
- declaration/body/name ranges；
- `entity_key` 与 anchor fingerprint。

但是：

- `query/signatures.rs` 从 `SymbolRecord` 重新拆参数；
- `query/hover.rs` 复用 definition ranker；
- `server/language_server.rs` 的 Definition 仍以 symbol records 为核心；
- `.h/.c` documentation pairing 再次对 symbol signature 做字符串 normalize。

这会让同一个函数在不同模块里拥有不同的 identity、arity 和配对判断。v1.4.2 需要把这几条读路径收敛到 callable facts。

### arity 已存在，但只被调用关系消费

`CallSiteFact.argument_count` 与 `SignatureShape::accepts_arity()` 已经能表示：

- 固定参数函数；
- 默认参数形成的 `min..=max`；
- variadic 的 `>= min`；
- `foo(void)` 的零参数；
- C 旧式 `foo()` 的 unknown arity。

调用关系会排除 `Some(false)`，但 Hover、Definition 和 Signature Help 没有复用这条证据。因此本次不再新写一个 signature comma counter 作为主判定，而是把现有 callable facts 接入共享候选集。

### `.h/.c` 目前不是一对一关系

当前 pairing 只要求：

- 同名；
- 同 kind；
- normalize 后 signature 相同；
- 双方有相同 `ProjectKey`。

它不要求 source 能 reach header，也不要求 header 只对应一个 source。其结果是多个 `.c` 可以在 Hover 阶段被投影成同一个 `.h`，而 anchor navigation 只是把某个对侧候选排到第一项，仍然返回全部 locations。

新设计需要把 pairing 表达为请求期二分图，并且只接受双向 degree 都为 1 的边。

### Record facts 没有进入 Hover

`RecordDef` 已经有 record specifier 的完整 `start_byte..end_byte`，但 `RecordDef.signature` 只保存 body 前缀。普通 struct 偶尔能在 Hover 中看到压成一行的 body，是 AST type symbol signature 的副作用，不是 record hover 合同。

因此完整定义必须按 record source range 读取原始文本，不能从压平 signature 或 members 重新拼装。

### Alias facts 缺少 declarator 形状

当前 `TypeAlias` 只描述终端方向：

- `RecordKey`；
- `NamedRecord { tag, kind }`；
- `UnresolvedTypeName`。

它没有表达 `*`、数组、qualifier、多 declarator 等形状。例如：

```c
typedef struct Foo *FooPtr;
typedef struct Foo FooArray[4];
```

两者都可能解析到 record `Foo`，但正确的 aka 分别是 `struct Foo *` 和 `struct Foo[4]`。如果只根据 terminal record 写 `(aka. struct Foo)`，会产生错误语义。

## 目标领域模型

### `CallSiteContext`

`CallSiteContext` 是对已有 `CallSiteFact` 的请求期投影，不另起持久化事实：

```rust
pub struct CallSiteContext {
    pub callee_name: String,
    pub form: CallForm,
    pub callee_range: SourceRange,
    pub argument_count: Option<u32>,
    pub argument_state: ArgumentState,
    pub reliability: ContextReliability,
}

pub enum ArgumentState {
    Complete,
    Partial { minimum_arity: u32, active_argument: u32 },
    Unknown,
}
```

其中：

- 完整且无 syntax-error overlap 的 call 可以给出精确 arity。
- Signature Help 正在输入的 call 只能给出 active parameter 和最小已知 arity。
- template-like comma、宏、解析错误等情况降为 unknown。

### `ArityCompatibility`

arity 兼容性使用三态，不使用 bool：

```rust
pub enum ArityCompatibility {
    Compatible,
    Incompatible,
    Unknown,
}
```

`Unknown` 不是低分的 `false`。它表示当前事实不足，必须继续保留候选，但可以排在 proven compatible 之后。

### `CallableCandidateSet`

`CallableCandidateSet` 是一次 exact-name 窄查询的 bounded working set。它持有已有 canonical types，而不是复制它们：

```rust
pub struct CallableCandidateSet {
    pub anchors: Vec<ResolvedCallableAnchor>,
    pub groups: Vec<CallableVariantGroup>,
    pub coverage: CandidateCoverage,
}
```

`ResolvedCallableAnchor` 至少组合：

- `CallableAnchor`；
- 从 anchor 投影的 `DefinitionCandidate`；
- full canonical signature；
- `SignatureShape`；
- `ScopeTier / confidence / reason`；
- arity compatibility；
- project/reach/linkage evidence；
- overlay/base provenance。

这只是 query DTO，不取代 `CallableAnchor`、`CallableEntity` 或 `DefinitionCandidate`。

### `CallableVariantGroup`

一个 group 表示可以安全视为同一逻辑函数的 anchors：

```rust
pub struct CallableVariantGroup {
    pub header_declaration: Option<CallableAnchor>,
    pub source_definition: Option<CallableAnchor>,
    pub other_variants: Vec<CallableAnchor>,
    pub group_tier: ScopeTier,
    pub counterpart_evidence: CounterpartEvidence,
}
```

`group_tier` 取 variants 中最强 scope evidence。展示时可以选 header，但不能因此丢掉 source 的 Current/Reachable 证据。

### `AliasResolution`

alias resolver 需要保留解析过程，而不是只返回 terminal record：

```rust
pub struct AliasResolution {
    pub alias: TypeAliasCandidate,
    pub chain: Vec<TypeAliasCandidate>,
    pub terminal_records: Vec<RecordCandidate>,
    pub winning_tier: Option<ScopeTier>,
    pub aka_spelling: Option<String>,
    pub status: AliasResolutionStatus,
}
```

`AliasResolutionStatus` 至少区分：

- `UniqueRecord`；
- `AmbiguousRecord`；
- `Unresolved`；
- `Cycle`；
- `UnsupportedDeclarator`；
- `Truncated`。

这样 Hover 可以只在 `UniqueRecord + exact aka spelling` 时显示 `(aka. ...)`，其余情况保留 typedef 自身而不硬猜。

### `CandidateCoverage`

唯一性依赖 coverage，因此 bounded query 需要携带：

```rust
pub struct CandidateCoverage {
    pub scanned: usize,
    pub truncated: bool,
    pub scope_open: bool,
    pub incomplete_reason: Option<CandidateIncompleteReason>,
}
```

首版不要求把完整 coverage 放进标准 LSP response，但纯 query 层必须保存并单测。debug 日志和 Hover fallback 可以选择性展示简短原因。

## schema 16

### 升级策略

v1.4.2 将 schema 从 `15` 升级到 `16`。升级采用现有 side-by-side generation rebuild：

1. 在独立 build 数据库写 schema 16 facts。
2. 创建 lookup indexes 并执行 `ANALYZE`。
3. 执行 `quick_check`、foreign-key check 和 WAL checkpoint。
4. 关闭 build connection。
5. 封装成新的 generation 文件。
6. 原子切换 `active-index` manifest。
7. 已持有 lease 的 schema 15 generation 允许请求读完，最后一个 lease 释放后再清理。

不提供 schema 15 到 16 的行级 migration，也不在生产查询中保留 `if old_schema` 分支。

### `callable_anchor_facts`

在现有 arity/range/linkage 字段基础上增加：

- `canonical_signature_id`：规范化完整声明签名，用于严格 declaration/definition equality。
- `presentation_signature_id`：不含 body、保留用户可读声明形态的签名。
- `signature_fidelity`：`ast_exact / lexical_fallback / malformed`。
- 必要时增加 canonical signature digest index，避免重复长字符串比较。

现有只表示 parameter list 的 `signature_id` 继续承载 `SignatureShape.normalized`，不能被复用为完整签名 identity。

canonical signature 的初始合同：

- 删除函数 body；
- 删除尾部 `;` 和纯格式差异；
- 规范空白与 `()[],*&` 周围间距；
- 保留返回类型、函数名、参数类型和参数名；
- 保留会改变 linkage/identity 判断的 storage 信息；
- 不执行 typedef 展开、宏展开或类型等价化。

这个合同故意保守。`int foo(int x)` 与 `int foo(int value)` 首版不自动判为完全一致，除非以后增加独立的 weaker compatibility evidence。

### `record_facts`

现有 record specifier range 继续保留，并补充：

- `body_start_byte / body_end_byte`；
- `declaration_start_byte / declaration_end_byte`；
- 对应 UTF-16 line/column；
- `range_fidelity`。

`declaration_range` 用于包含 record 后的 `;`，以及在语法树可以证明时包含直接所属的 declaration wrapper。record 本体和 typedef wrapper 仍然是不同事实，不能把多个 alias 混成一个 record 名。

### `type_alias_facts`

增加：

- alias declaration 的完整 byte/position range；
- `underlying_spelling`；
- `declarator_shape`；
- `target_fidelity`；
- 每个 declarator 独立的 stable fingerprint。

`declarator_shape` 首版至少支持：

- `Identity`；
- `Pointer { qualifiers }`；
- `Array { extent_text }`；
- `Qualified`；
- `Unsupported`。

多 declarator typedef 必须为每个 alias 生成独立 shape。例如：

```c
typedef struct Foo Foo, *FooPtr;
```

应产生两个 alias facts，而不是共享一个丢失 declarator 的 target。

`typedef struct Foo Foo;` 即使 alias 名与 tag 名相同，也必须保留 alias fact。cycle detection 改为基于 alias fact fingerprint/id，而不是仅凭 name，因此不会把这个合法关系误当作 self-cycle。

### parser version 与重建

- parser fact version 从硬编码值迁移为命名常量。
- schema 16 同时推进 parser fact version。
- schema mismatch、parser fact version mismatch 都触发全量 rebuild。
- 不尝试让旧 record/alias rows 猜测新增 declarator shape。

## 协议决策

### 标准 LSP surface 不新增私有替代品

本次能力可以由标准协议表达：

- Hover 使用 `textDocument/hover`；
- 多签名使用 `textDocument/signatureHelp`；
- 跳转使用 `textDocument/definition`；
- 多个 definition 候选继续返回 `Location[]`。

因此不新增 `fossilsense.smartHover` 或 `resolveDefinition` 之类平行协议。

### 允许改变标准请求的行为

虽然 wire shape 不变，但返回语义会发生破坏性变化：

- 已证明一对一的 declaration/definition anchor navigation 只返回对侧一个 location。
- call-site definition 会先应用 arity compatibility。
- Signature Help 会去除 `.h/.c` 重复 variants，并改变 `activeSignature`。
- Hover 会对逻辑函数 group 只显示 header representative。

不提供旧行为配置开关。

### Completion resolve data 升级

函数 completion documentation 需要复用新的 callable group，因此 `CompletionItem.data` 的内部 payload 增加显式版本，例如 `v2`：

- server 只生成新 payload；
- extension 与 server 同版本打包；
- 无法识别的旧 payload 直接返回无 documentation 的原 item，不尝试猜字段；
- 不保留旧 payload builder。

### Call Relations v2

`fossilsense.lsp.callRelations` 的 wire shape 本次不需要改变，因此保持 v2。若 callable identity/counterpart resolver 的版本进入 cursor：

- 推进 cursor 中的 resolver version；
- 旧 cursor 明确失效并要求从第一页重新查询；
- 不提供跨 resolver version continuation。

这不是为了兼容旧 cursor，而是防止新旧候选关系混页。

## 函数候选共享管线

### 召回

函数 exact-name 查询使用 `CallFactStoreView::anchors_by_names()` 或更窄的单 name API，不再先查全部 `SymbolRecord` 再尝试和 callable anchors 对齐。

召回需要合并：

- generation-pinned base anchors；
- dirty open-document anchors；
- 当前请求文档 live parse；
- external header declaration anchors。

同路径存在 overlay 时，base rows 必须被 shadow。合并后按 anchor fingerprint 去重。

### arity 筛选合同

完整 call 的筛选顺序：

1. `SignatureShape::accepts_arity(argument_count) == Some(true)` 标为 Compatible。
2. `None` 标为 Unknown，继续保留。
3. `Some(false)` 标为 Incompatible。

返回政策：

- 存在 Compatible 时，删除 Incompatible，Unknown 排在 Compatible 后。
- 没有 Compatible 但存在 Unknown 时，只保留 Unknown。
- 全部为 Incompatible 时，恢复原候选作为 `arity_mismatch_fallback`，避免参数写错时完全失去 Hover/跳转；该 fallback 不得标成 exact binding。
- call context 不可靠时不执行 hard filter。

Signature Help 的 partial call 使用 minimum arity：

- 只能排除 `max_arity < minimum_arity` 的候选；
- `foo(` 且尚未输入实参时，minimum arity 为 0，而不是 1；
- 每出现一个确定属于当前 argument list 的顶层逗号，minimum arity 才增长；
- template-like comma、initializer、字符串和嵌套调用不能污染计数。

### 排序合同

arity compatibility 是 match evidence，不替代 `ScopeTier`：

```text
ScopeTier
    -> arity compatibility
    -> feature-specific role preference
    -> locality
    -> path/range stable tie-break
```

如果某功能需要让 Compatible 跨 tier 反超，需要单独写出合同；v1.4.2 默认不允许低 tier 仅凭 arity 反超高 tier candidate。arity 首先用于排除已证明不兼容者。

## `.h/.c` counterpart resolver

### 文件角色

header family 沿用现有扩展集合：

- `.h`
- `.hh`
- `.hpp`
- `.hxx`
- `.inl`

source family：

- `.c`
- `.cc`
- `.cpp`
- `.cxx`

### 边的建立条件

对于同一次 exact-name 查询，把 source definitions 和 header declarations 构成二分图。只有满足以下全部条件才建立候选边：

1. `name` 相同。
2. callable kind 相同，首版正式支持 free function。
3. source anchor role 是 definition，header anchor role 是 declaration。
4. canonical full signature 完全相同。
5. linkage 允许跨文件；不同 internal linkage domain 不连边。
6. header 位于从 source path 出发的现有 bounded include closure 中。
7. source reach scope 是 closed；unresolved、ambiguous、DepthLimit、NodeLimit 时不证明唯一 pairing。
8. 候选召回未截断。

`ProjectKey` 不再作为硬门槛。原因是一个 source 可以合法 include 共享项目头文件，两个文件的最近 marker 不同并不能否定声明/定义关系。ProjectKey 只可以作为同分 evidence 或 debug 解释，不能单独建立配对。

### 唯一配对

候选边建立后，只接受以下边：

```text
degree(source) == 1
&& degree(header) == 1
```

这意味着配对是双向唯一的。如果一个 header 被两个平台 source definitions 共同实现，则 `.h -> .c` 保持多个候选，不直接跳。

### Hover policy

未证明同一逻辑函数时：

- `ScopeTier` 继续严格主导。
- 同 tier 内 header declaration 排在 source definition 前。
- 不因为扩展名相同就去重。

证明为唯一 group 后：

- group 只产生一个 Hover section。
- 可见 representative 固定为 header declaration。
- path、signature、attached documentation 以 header 为准。
- group tier 继承 variants 中最强 tier。
- source-only comment 不自动合并到 header，避免形成无法解释的混合文档；以后如需 fallback，必须单独立合同。

### Definition policy

光标位于 declaration/definition anchor：

- 唯一 paired source definition -> 只返回 header declaration。
- 唯一 paired header declaration -> 只返回 source definition。
- 无唯一 pair -> 返回正常排序候选，不伪装直跳。

光标位于普通 call site：

- 先应用 arity 筛选。
- group 内 source definition 作为主要跳转目标。
- 没有 source body 时才返回 header declaration。
- 多个逻辑 groups 时继续返回 `Location[]`。

### Signature Help 与 Completion Documentation

- Signature Help 每个 logical group 只显示一个 signature，首选 header presentation。
- `activeSignature` 从最高 tier 的 Compatible group 中选择。
- 同一 group 的 `.h/.c` 不占两个签名槽位。
- 函数 completion documentation 使用相同 group 和 header representative，不能继续走旧的 ProjectKey-only documentation ranker。
- Call Hierarchy 继续以 source body 为 primary anchor，但 variants 和 hover presentation 使用同一 counterpart group，不另行配对。

## Struct 完整 Hover

### 触发与候选

当 hovered token 可以对应 record tag、typedef name 或 record type use 时，查询 `RecordCandidate`：

- 使用 current path 与 `ReachScope` 计算 tier。
- dirty current document 的 record facts shadow base rows。
- 按 record id/fingerprint 去重。
- 同一最高 tier 有多个 record 时可以显示多个 section，但不能把第一项叫作唯一类型。

### 定义展示

完整定义来自 record source range，而不是 compact signature：

```c
struct Packet {
    uint16_t type;
    uint16_t length;

#ifdef ENABLE_TIMESTAMP
    uint64_t timestamp;
#endif
};
```

需要保留：

- 原始换行和缩进；
- 字段、方法和嵌套 record；
- body 内注释；
- 条件编译文本；
- 可以被 range 覆盖的结尾分号。

不通过 members 重新生成源码。members 只用于成员补全和辅助结构信息。

### SourceExcerptReader

新增协议无关的 bounded range reader：

- 当前 buffer 按同一 version 的 byte range 切片。
- 磁盘文件使用 seek/range read，不因整个文件超过 256 KiB 就拒绝一个很小的 struct。
- 读取前校验 active revision 的 size/mtime；不匹配时视为 stale range。
- 单个 record definition 设置独立 byte/line budget。
- 超预算不静默截成“看似完整”的定义，而是显示明确的 omitted/truncated 提示并退回 record signature。

首版建议预算：

- 单 definition 最大 `128 KiB`；
- 最大 `2048` 行；
- Hover 总 Markdown 继续受总候选数和总字符预算限制。

这些常量必须集中定义并单测，不能散落在 server handler。

### 注释和 evidence

- record 前置/同行注释继续走共享 comments pipeline。
- 完整定义代码块与文档 Markdown 分离。
- path、tier、confidence、reason 保留。
- range stale、文件不可读或 AST fact unavailable 时降级为 signature-only，并保留候选 evidence。

## typedef Hover 与 `aka`

### 展示顺序

typedef Hover 先保留 alias 本身，再展示终端 record：

```text
PacketT (aka. struct Packet)

typedef struct Packet PacketT;

struct Packet {
    int size;
};
```

这意味着 alias documentation 不会因为展开 record 而消失。record documentation 与 alias documentation 分为两个明确 section，不能混成一个无法判断归属的注释块。

### 唯一性合同

显示 `aka` 必须满足：

1. alias chain 解析完成且没有 cycle。
2. terminal target 是 record。
3. 按 record id/fingerprint 去重。
4. 只考虑最高 `ScopeTier` 后恰好一个 terminal record。
5. query coverage 未截断。
6. `aka_spelling` 可以由 declarator shape 精确生成。

如果存在一个 Reachable record 和一个无关 Global record，Reachable 可以成为唯一 winning target。如果同 tier 有两个 record，则不显示 aka。

### declarator shape

支持准确输出时：

- `typedef struct Foo FooT;` -> `(aka. struct Foo)`
- `typedef struct Foo *FooPtr;` -> `(aka. struct Foo *)`
- `typedef const struct Foo FooConst;` -> `(aka. const struct Foo)`
- `typedef struct Foo FooArray[4];` -> `(aka. struct Foo[4])`

alias chain 只有在 shape 能可靠组合时才继续生成最终 aka。函数指针、复杂括号 declarator 或 parser error 区域标为 `UnsupportedDeclarator`：

- 可以继续展示 typedef 原声明；
- 如果 terminal record 唯一，可以另设“underlying record”section；
- 不能输出不准确的 `(aka. ...)`。

匿名 typedef struct 使用：

```text
Buffer (aka. anonymous struct)
```

并展示完整 typedef declaration/body。如果未来决定省略 anonymous aka，只改 presentation policy，不改变 alias/record facts。

### cycle 与同名 tag

- alias cycle 按 alias fact id/fingerprint 防环。
- `typedef struct Foo Foo;` 同时存在 tag `Foo` 与 ordinary identifier namespace 的 alias `Foo`，不能因字符串同名跳过 alias fact。
- 解析 `struct Foo` 上的 `Foo` 时显示 record tag；解析 bare `Foo` type use 时可以显示 typedef identity。无法取得 occurrence role 时保留 best-effort 候选，不伪装 namespace binding。

## dirty-document overlay

### 统一 overlay 语义

当前 Hover、Definition、completion 和 call relation 对 open documents 的覆盖程度不一致。v1.4.2 引入共享的 candidate overlay view：

- 只纳入内容与 active revision hash 不同，或已保存但尚未发布匹配 revision 的文档。
- dirty path 完全 shadow 同路径 base callable/record/alias facts。
- exact-name 请求只从相关 overlay name index 取候选，不遍历所有源码文本。
- overlay generation 覆盖同 workspace 全部有效 dirty documents，因此修改另一个已打开 header 后，当前 source 的 Hover/Signature Help 可以看到新声明。
- latest revision wins；过期 parse 结果不能覆盖新 document version。

### parse facts

不新增另一套 parser 入口。为调用方定义组合 mask，例如：

```rust
const HOVER_SEMANTICS =
    ParseFacts::SYMBOLS
    | ParseFacts::RECORDS
    | ParseFacts::ALIASES
    | ParseFacts::CALL_RELATIONS;
```

Definition、Hover 和 Signature Help 可以根据实际需要请求其子集或并集，但仍通过 `parse_with_handle` / document singleflight 获取同一 `FileSemanticIndex`。

### publication 与失效

- dirty document version 变化推进 overlay epoch。
- include 变化生成新的 dirty `ReachGraph`，不原地修改已发布 graph。
- 请求 memo key 必须包含 engine epoch、semantic generation、overlay epoch、document version 和 query kind。
- 文件 active revision hash 与保存内容匹配后才清除 overlay。

## 降级合同

| 缺失证据 | 行为 |
|---|---|
| 没有 call context | 按 exact-name 与 scope 候选工作，不做 arity 筛选 |
| call arity unknown | 保留所有候选，Signature Help 只标 active parameter |
| 所有函数都 arity incompatible | 恢复候选作为 mismatch fallback，不能返回伪 exact |
| ReachGraph unavailable | 不建立新 `.h/.c` 唯一 pair，保留普通排序 |
| source reach scope open | 不证明 counterpart uniqueness |
| 配对图一对多/多对一 | 不折叠，不直接跳 |
| callable facts lexical fallback | 回退 symbol candidate，标记低 fidelity |
| record facts unavailable | 回退通用 type signature Hover |
| record range stale/unreadable | 不读取错误片段，回退 signature-only |
| alias unresolved/cycle | 显示 typedef 自身，不显示 aka |
| alias terminal record 同 tier 歧义 | 展示候选或 typedef 自身，不显示 aka |
| declarator shape unsupported | 不伪造 aka，可显示 underlying record section |
| exact-name query 达到 scan cap | 标记 truncated，不证明唯一关系 |

降级不能使整个 Hover 或 Definition 因一个增强事实缺失而消失。反过来，也不能为了“总有结果”把不完整 evidence 写成确定关系。

## 请求预算与大工作区约束

### exact-name query

- callable、record 和 alias durable reads 都必须有 name/path 索引。
- 请求只物化 exact-name 命中的 bounded rows。
- pairing uniqueness 需要看到完整 relevant set；一旦达到 scan cap，就停止证明 unique pair，而不是在截断集合内宣布唯一。
- 建议 callable/record/alias raw scan cap 分开定义，初始值通过 `samples/mini-c`、HimuOS 和大工作区基准校准。

### hydration

- 先筛选和归并，再读取 source excerpt。
- Hover 最多 hydrate 最终可见 sections。
- Signature Help 只为返回签名读取必要 comment，不读取 record body。
- Definition 不读取源码正文。
- Completion 热路径不受影响；documentation 继续在 resolve 阶段读取。

### ReachGraph

- counterpart resolver 对每个 source candidate 调用 memoized `ReachGraph::reachable(path)`。
- 不建立 workspace 全量 `.h/.c` pairing catalog。
- dirty include graph 与 request snapshot 绑定。
- reach closure 达到 depth/node limit 时，candidate 可以继续参与普通排序，但不参与唯一 pair 证明。

### 可观测性

verbose/perf 日志只输出：

- raw/filtered/grouped/returned candidate count；
- arity compatible/unknown/incompatible 聚合数；
- counterpart edge/group/ambiguous 数；
- record/alias hydration count 和字节数；
- query/hydration/reach 各阶段耗时；
- coverage/truncation/fallback reason。

默认日志不输出 symbol name、alias spelling、源码片段或用户路径。

## 分步实施与验收

## Step 0：固定新行为合同与架构 fitness

### 实施

在修改 production code 前增加失败测试，固定以下跨功能合同：

- 同一个 call-site arity context 对 Hover、Definition 和 Signature Help 产生同一组 logical callable candidates。
- `.h/.c` 只有双向唯一时才折叠和直跳。
- Hover 的 header-first 不改变普通 call-site Definition 的 source-first。
- record/alias uniqueness 只看 winning tier，并受 coverage/open scope 约束。
- dirty path shadow base facts。
- schema 16 不允许新增 server 直写 SQL、query 依赖 `rusqlite` 或 `tower-lsp`。

新增 fitness checks：

- `server/hover.rs` 不直接调用 `symbols_by_name()` 处理函数候选。
- `server/signature_help.rs` 不再从 `SymbolRecord` 重新拆函数 arity。
- `query/documentation.rs` 不保留 ProjectKey-only `.h/.c` pairing 生产入口。
- record/alias durable reads 只能通过 typed view。
- source excerpt reader 不允许无条件 `read_to_string` 整个候选文件。

### 验收

- 新测试在旧实现上失败，且失败原因对应本计划行为。
- architecture fitness 明确指出旧函数查询链的依赖位置。
- `cargo test -p fossilsense` 的其它基线失败必须先解释，不能在大改后混在一起处理。

## Step 1：扩展 parser facts 与 schema 16

### 实施

扩展 callable、record 和 alias facts：

- callable anchor 产生 presentation signature 与 canonical full signature。
- record 产生 specifier/body/declaration ranges 与 fidelity。
- alias 为每个 declarator 产生 declaration range、underlying spelling、shape、fingerprint。
- 保留 `typedef struct Foo Foo;` 的 alias fact。
- complex declarator 无法可靠建模时显式输出 `Unsupported`，不能回退成 `Identity`。

更新 persistence：

- schema version -> `16`。
- parser fact version 使用命名常量并推进。
- typed write DTO 增加新字段。
- `store_parser_adapter` 仍然是 parser product 到 persistence port 的唯一适配器。
- 新 indexes 覆盖 callable exact name/canonical signature、alias exact name 和 record tag/typedef names。
- schema mismatch 只走 full rebuild。

### 验收

Parser 测试至少覆盖：

- C/C++ free function declaration/definition canonical signature parity。
- whitespace、body、尾分号不影响 canonical signature。
- 参数名不同不会被误写成完全一致。
- fixed/default/variadic/`void`/C unspecified arity。
- named struct、anonymous typedef struct、union/class ranges。
- record 内注释和 preprocessor 文本仍落在 excerpt range。
- identity/pointer/qualified/array/multi-declarator typedef。
- alias 与 tag 同名。
- function pointer 或 malformed declarator 标为 Unsupported。

Store 测试至少覆盖：

- schema 16 tables/views/indexes。
- 新 typed rows round-trip。
- schema 15 fixture 触发 rebuild，而不是被兼容读取。
- generation side-by-side publication 和 lease cleanup 不回退。
- active views 只暴露 active revision 的新 facts。

## Step 2：建立 generation-pinned CandidateQueryService 与 overlay

### 实施

建立统一请求输入：

- `RequestContext`；
- current URI/path/position；
- document version 与 text snapshot；
- generation-pinned callable/record/alias views；
- ReachGraph；
- workspace dirty-document overlay generation；
- query budget/cancellation。

建立 exact-name overlay indexes：

- callable anchors by name；
- records by display/tag/typedef name；
- aliases by alias name；
- 路径 shadow set。

base query 与 overlay merge 必须遵循：

1. 读取同 generation base narrow rows。
2. 跳过所有 shadowed paths。
3. 合并相关 overlay facts。
4. 按 stable fingerprint/id 去重。
5. 生成 coverage。

### 验收

- 当前文档新增/修改/删除函数会即时影响三种函数查询。
- 修改另一个已打开 header 后，source 中的 Signature Help 和 Hover 使用新声明。
- dirty record/alias shadow 旧 DB range，不出现新旧两份 definition。
- 保存但索引尚未发布时 overlay 不提前清除。
- active revision hash 匹配后 overlay 才退出。
- request 中途 publication 不混入新 generation。
- cancellation 不发布部分 memo，也不污染下一请求。

## Step 3：接入 CallSiteContext 与 arity pipeline

### 实施

完整 call 使用 parser-produced `CallSiteFact`：

- position 必须落在 callee range。
- syntax-error overlap 降为 unreliable。
- direct/qualified/parenthesized name 是正式支持范围。
- member/function-pointer/callable-object 保留事实但不伪装成 free-function binding。

partial Signature Help 改进 `call_context_at`：

- 返回 `ArgumentState`，而不只返回 `active_argument`。
- 区分空参数列表与已输入第一参数。
- 保持 string/char/comment/nested delimiters 规则。
- template-like 无法判断时降为 Unknown。

实现共享 `ArityCompatibility` 和 filter policy，Definition/Hover/Signature Help 只消费结果。

### 验收

纯 query 测试：

- 0/1/2 固定参数。
- 默认参数 `min..=max`。
- variadic `>= min`。
- C `foo()` unknown 与 `foo(void)` exact zero 的区别。
- complete call 的 exact count。
- partial call 的 minimum count。
- nested call、initializer、string comma、comment comma。
- all-incompatible fallback。
- compatible + unknown + incompatible 的稳定顺序。
- scope tier 仍然主导，不因 arity 引入 magic score。

LSP 测试：

- `pick(1, 2)` 的 Hover 和 Definition 不返回已证明只接受一个参数的候选。
- Signature Help 仍可展示多个 arity-compatible 签名并正确设置 active parameter。
- 参数暂时写错时仍可跳转，并体现 fallback 而不是返回空。

## Step 4：建立唯一 `.h/.c` counterpart graph

### 实施

在 `query/callables/counterpart.rs` 中实现请求期二分图：

- 输入是 arity 处理后的 same-name callable anchors。
- 对每个 source definition 读取 memoized source reach scope。
- 只有 strict signature/linkage/reach/coverage 全满足才连边。
- 对双向 degree=1 的边生成 `CallableVariantGroup`。
- 其它边保留 ambiguity evidence，但不折叠。

更新 consumers：

- Hover：group header representative。
- Definition anchor：唯一对侧单 location。
- Definition call site：source definition primary。
- Signature Help：group 去重，header presentation。
- completion documentation：删除旧 ProjectKey-only pairing。
- Call Hierarchy：验证 source-body primary 与 group variants 一致。

删除旧实现：

- 旧 `rank_documentation_candidates()` 中承担 identity 的逻辑。
- `same_documentation_project()` 作为 pairing hard gate。
- 仅通过排序把 counterpart 放首项的旧 anchor toggle。
- 为保持旧重复 Hover section 而存在的去重补丁。

ProjectKey 如果保留，只作为 evidence 字段和稳定 tie-break，不能重新成为 hard binding。

### 验收

至少覆盖以下图形：

```text
一对一：        h1 <-> c1       => pair
一对多：        h1 <-> c1,c2    => no direct pair
多对一：        h1,h2 <-> c1    => no direct pair
签名不同：      h1    c1        => no edge
不可达：        h1    c1        => no edge
open reach：    h1 ?  c1        => no unique proof
internal：      h1    c1        => no cross-file pair
```

LSP smoke 必须断言：

- paired Hover 中 `.c` section 完全缺席，不只是 `.h` 出现在前面。
- `.c` anchor Definition response 只有 `.h` 一个 location。
- `.h` anchor response 只有 `.c` 一个 location。
- 普通 call site 仍以 `.c` 为第一目标。
- 无 project marker 但有唯一 closed reach edge 时仍可配对。
- 同 ProjectKey 但不可达时不再配对。
- 多实现文件时 header Definition 返回多个候选而不是直跳。

## Step 5：重做函数 Hover 与 Signature Help presentation

### 实施

函数 Hover 消费 `CallableVariantGroup`：

- group 内 header-only presentation。
- 非 group 候选同 tier header-first。
- path/signature/comment/evidence 使用同一 chosen presentation anchor。
- 同 logical group 不重复 section。
- call site 存在多个 viable groups 时，Hover 可以附加简短提示，说明 Signature Help 可切换多个签名，但不能尝试从 server 主动触发客户端 UI。

Signature Help：

- 每个 group 产生一个 `SignatureInformation`。
- `activeSignature` 选择最高 tier 的 Compatible group。
- 没有 Compatible 时按 Unknown/fallback policy 选择。
- active parameter 必须在该 signature 范围内，否则为 `None`。
- Doxygen/XML parameter docs 仍按最终 presentation signature 映射。
- h/c comments 不混合。

### 验收

- 多候选可由 VS Code 原生上下切换。
- `.h/.c` 相同签名只出现一次。
- 同名同 arity不同签名继续分别显示。
- active signature 与 Hover/Definition 使用相同 logical group ordering。
- header parameter 名与 source 参数名不一致且 strict signature 不配对时，不错误挂载 parameter docs。
- unreadable comment source 只降为 signature/evidence，不影响签名列表。

## Step 6：接入 Record-aware Hover

### 实施

- `MemberStoreView` 增加 Hover 需要的 typed narrow reads，或抽出不带 completion policy 的 record read view。
- query 层按 current path/reach/overlay 构造 `RecordCandidate`。
- `SourceExcerptReader` 读取 record declaration/specifier range。
- 保留原始多行定义。
- comments pipeline 使用明确 record declaration anchor。
- generic type symbol Hover 只作为 record facts 不可用时的 fallback。

需要注意：成员补全当前的 `resolve_record_candidates()` 包含 alias recursion policy。Hover 可以复用 canonical record candidates 和 tier 规则，但不能把“成员可访问的 record”直接等同于“alias Hover 唯一 target”，因此 alias trace 必须由独立 typed DTO 保留。

### 验收

- named multiline struct 展示完整 body、注释和 `#ifdef`。
- union/class 通过同一 reader 工作，不破坏 struct 主验收。
- 当前未保存 record 立即显示新字段。
- 外部大头文件中小 record 不因整文件超过 256 KiB 而丢失。

## 端到端验收样例

### arity 筛选与 Signature Help

```c
// one.h
int lookup(int key);

// two.h
int lookup(int table, int key);

// main.c
int value = lookup(3, 7);
```

预期：

- 悬停 call-site `lookup` 不显示单参数候选。
- Definition 不返回单参数候选。
- Signature Help 显示仍可接受两个参数的 logical groups。
- 如果只有一个两参数 group，`activeSignature = 0`。

### 唯一 `.h/.c`

```c
// api.h
int read_packet(int fd);

// api.c
#include "api.h"
int read_packet(int fd) { return fd; }
```

预期：

- 两边 canonical signature 相同。
- `api.c` 的 closed reach scope 唯一包含 `api.h` 匹配声明。
- Hover 只显示 `api.h`。
- 在 `api.c` definition 上跳转只到 `api.h`。
- 在 `api.h` declaration 上跳转只到 `api.c`。
- 普通调用点跳转首先到 `api.c`。

### 多实现不直跳

```text
api.h
├── api_windows.c
└── api_linux.c
```

如果两个 source 都有相同 canonical signature 并 reach `api.h`，header degree 为 2：

- Hover 可以保留分离候选或按 scope 排序，但不能建立唯一 group。
- `.h` Definition 返回两个 source candidates。
- UI 不显示“唯一 counterpart”类提示。

### Struct 完整 Hover

```c
struct Packet {
    int type;
    /* byte count */
    int size;
#ifdef WITH_CHECKSUM
    int checksum;
#endif
};
```

预期 Hover 代码块保留字段注释、换行和 `#ifdef`，而不是：

```c
struct Packet { int type; int size; int checksum; };
```

### typedef + aka

```c
struct Packet {
    int size;
};

typedef const struct Packet *PacketView;
```

预期：

```text
PacketView (aka. const struct Packet *)
```

随后分别显示 typedef declaration 和完整 `struct Packet`。

如果同一 winning tier 存在两个 `struct Packet`，则只显示 typedef 自身和歧义候选，不显示 aka。

## 风险与控制

### strict signature 可能漏配

header/source 参数名不同、storage spelling 不同或宏参与声明时，strict canonical signature 可能无法配对。这是有意的保守结果：漏配会退回多个候选，错配则会造成错误直跳和错误 Hover 折叠。

以后若要扩大召回，可以新增 weaker signature compatibility evidence，但不能放松唯一 direct-jump 的 strict gate，除非有新的独立证据。

### open reach scope 会减少 pairing

大型老仓库经常有 unresolved include，因此要求 closed scope 会让一部分现有同项目配对消失。这里优先保证“直跳意味着有一对一证据”。普通 header-first Hover 排序和候选导航仍然可用，因此这是可解释降级，不是能力整体消失。

### alias declarator 容易滑向完整类型系统

本计划只解析生成准确 aka 所需的有限 declarator shape。遇到函数指针、复杂括号或模板 alias 时输出 Unsupported。不能因为 schema 已经增加 shape 就继续扩展到完整 C/C++ 类型等价系统。

### source range 可能 stale

索引发布到文件 watcher 响应之间存在窗口。range reader 必须校验 revision metadata，dirty document 必须 shadow base。校验失败时 signature-only 比读取错误字段更好。

### exact-name 候选可能很多

全局常用名字可能产生大量 anchors。查询达到 cap 时可以返回 top candidates，但 uniqueness 相关功能必须关闭并标记 truncated。不能先截断再宣布 degree=1。

### 行为变化可能影响用户习惯

paired anchor Definition 从“对侧排第一但仍有列表”变为“直接单跳”，Signature Help 也会去掉重复 `.h/.c`。这是本版本明确的体验目标，不提供兼容选项。Release note 需要写清普通 call-site 与 anchor navigation 的区别。

## 明确删除项

完成本计划时，下列旧设计不应继续作为 production truth：

- 函数 Hover/Signature Help 使用 `rank_definition_records_with_scope(SymbolRecord)`。
- `.h/.c` 仅凭同 ProjectKey + compatible signature 建立 documentation pairing。
- counterpart navigation 只通过 stable sort 把对侧放到第一项。
- Hover 在 presentation 后按 path/line/signature 去重来模拟实体归并。
- alias resolver 丢弃 alias trace 后由 server 猜 `aka`。
- record Hover 依赖压平 type symbol signature 作为“完整定义”。
- 为 schema 15 保留生产双读。

如果某个旧 helper 仍被非目标功能需要，应拆出其真正通用部分并重新命名，不能整块保留旧 policy。
