# FossilSense 1.4.0 Phase 5 generation lease and cleanup benchmark

status: current

measured_at: 2026-07-12

## Scope

This phase closes the side-by-side publication lifecycle. `CallReadHandle` now owns a cloneable `IndexDbLease`. Publication and startup cleanup preserve the active manifest target and every leased generation; the final lease drop retries cleanup, so Windows readers finish before file removal. A missing, corrupt, or missing-target manifest suppresses generation deletion, preserving force-rebuild recovery candidates. Build and manifest staging older than 24 hours are eligible for startup cleanup.

## Synthetic family result

Release-mode diagnostic on NTFS:

| Metric | Result |
|---|---:|
| Old generation files | 1,000 |
| Leased old generations | 1 |
| Files removed in publication-style pass | 999 |
| Cleanup elapsed | 244.213 ms |

This deliberately stresses a family much larger than steady state. Once cleanup is active, ordinary publication scans only the active file, a leased predecessor while requests remain, and at most a small number of failed-deletion/staging artifacts. Cleanup is best effort and does not turn a successful manifest switch into failure.

## Correctness boundary

- The first publication lease keeps the old generation present across the second manifest switch; dropping its final lease removes it while preserving the active generation.
- Active, leased, unleased-old, and staging files are tested independently.
- Invalid/traversing/missing-target manifests preserve recoverable generation files.
- Temp cutoff is explicit; recent staging is not deleted by the 24-hour production cutoff.
- The existing side-by-side full-build test still verifies an already-open old SQLite reader across publication.
