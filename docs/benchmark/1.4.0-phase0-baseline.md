## Method

The release binary was measured with `scripts/benchmark_large_workspace.ps1 -Repeats 3`. Each case used a fresh process. Working Set and Private Bytes were sampled every 20 ms. The generated report contains anonymous case IDs and numeric counters only.

Correctness gates completed before measurement:

- 652 Rust tests passed; one diagnostic release benchmark remained ignored by default.
- 2 LSP smoke tests passed.
- Focused call-catalog tests passed without behavior changes.

## Catalog phase split

Representative Wine medium fan-in run:

| Phase | Time |
|---|---:|
| SQLite anchor load | 1.54–1.61 s |
| SQLite call-site load | 5.68–5.79 s |
| Entity grouping | 0.78–0.80 s |
| Candidate expansion / relation grouping | 0.69–0.72 s |
| Relation/adjacency finalization | 0.67–0.70 s |

The phase split confirms that Phase 1 must remove both the full SQLite fact load and the global derived relation structures. Optimizing only final DTO materialization cannot reach the cold-query or memory gate.

## SQLite tuning decision gate

`wal_autocheckpoint`, transaction sizing, deferred indexes, `synchronous`, and `temp_store` remain A/B variables rather than accepted fixes. Their full-rebuild experiments run against dedicated `target/benchmark/*-rebuild.sqlite` files and must record elapsed, peak memory, WAL/database bytes, and correctness together. No tuning value is promoted into the product from this baseline alone; Phase 3 owns the final choice after the side-by-side writer makes the comparison meaningful.
