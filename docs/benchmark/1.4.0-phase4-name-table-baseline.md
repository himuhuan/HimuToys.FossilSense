# FossilSense 1.4.0 Phase 4 NameTable baseline

status: current

measured_at: 2026-07-12

database: schema-15 Wine full index (local, generated)

## Scope

This release-mode diagnostic measures the current pre-Phase-4 `NameTable`: SQLite first materializes a `Vec<NameTableSymbolRow>` with owned strings, cold construction converts all rows to `Arc<str>`, re-interns paths/names/lower-names, builds a global sorted index and project postings, and computes all-workspace reach. Dirty replacement clones every unchanged entry and repeats all interning/sorting.

The dirty case selects the file with the most name rows without printing its path or symbols. It contains 14,173 of the Wine database's 495,645 rows and is the worst-case single-file publication gate.

## Baseline

Two adjacent release runs:

| Metric | Run 1 | Run 2 with memory sampling | Phase 4 gate |
|---|---:|---:|---:|
| Typed-row SQL load | 512 ms | 496 ms | diagnostic |
| Cold `NameTable` build | 862 ms | 814 ms | <800 ms target: marginal miss |
| Dirty replacement median | 1,064.485 ms | 979.272 ms | <100 ms: fail |
| Dirty Private Bytes delta | — | 101.625 MiB | <32 MiB: fail |

The dirty metric is the median of five identical replacements after the base table is built. A 1 ms sampler records process Private Bytes while old and new immutable tables coexist. The result establishes both the time and memory failure; Phase 4 must remove full-entry cloning, global re-interning, global sorting, and per-entry path duplication from single-file publication.

## Reproduction

```powershell
$env:FOSSILSENSE_BENCH_DB = (Resolve-Path 'target/benchmark/index-wine-rebuild.sqlite').Path
cargo test --release -p fossilsense `
  query::tests::benchmark_large_name_table_build_and_dirty_update -- `
  --ignored --exact --nocapture
```

The test is ignored in ordinary CI and fails fast unless `FOSSILSENSE_BENCH_DB` points to a prepared schema-15 database.
