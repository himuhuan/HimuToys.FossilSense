# FossilSense 1.4.0 Phase 3 side-by-side publication benchmark

status: current

measured_at: 2026-07-12

workspace: Wine, 7,153 files

## Scope

This measurement validates reader-safe full-build publication for the default workspace cache. A full, forced, or schema-mismatch rebuild writes a unique unpublished database, seeds its semantic generation from the previous active family, writes facts with deferred call indexes, validates and checkpoints the closed result, renames it to an immutable generation file, then atomically replaces the `active-index` manifest. Dirty indexing continues against the manifest's active WAL database.

Windows publication uses `MoveFileExW` with replace-existing and write-through flags. The generation database is renamed before the manifest switch, so a publication failure cannot expose a partial database. Old generation files are retained because older engine snapshots or SQLite readers may still hold them; bounded deferred cleanup remains release-hardening work.

## Wine default-cache measurement

The actual default cache resides on the system cache volume, whereas the Phase 3 schema/deferred-index A/B databases reside under the repository benchmark directory on another volume. The raw totals are therefore not a cross-volume speed comparison.

| Metric | Default side-by-side run |
|---|---:|
| `write_ms` | 49.268 s |
| `secondary_index_ms` | 7.984 s |
| `publication_ms` | 9.413 s |
| `elapsed_ms` | 82.396 s |
| measured wall time | 83.286 s |
| total DB | 355.285 MiB |

An explicit no-publication control in the same cache directory took 96.081 s, including 70.789 s of fact writes. This reversal demonstrates substantial system-volume/cache/antivirus variability, so it is not used as evidence that manifest publication accelerates full builds. The same-volume F: A/B in the deferred-index benchmark remains the evidence for the writer speedup.

Publication cost was isolated on the completed database:

| Gate | Warm isolated time |
|---|---:|
| `PRAGMA quick_check(1)` | 4.448 s |
| `PRAGMA foreign_key_check` | 1.661 s |
| WAL truncate checkpoint | 0.079 s |

The cold in-process aggregate was 9.413 seconds. This is an explicit safety cost, not hidden in `write_ms`; later release work may reduce it only with an equally strong corruption/publication gate.

## Effectiveness and correctness gates

- A Windows integration test held the old SQLite reader open while publishing generations 1 and 2. The old reader continued to return generation-1 content; manifest resolution and new readers returned generation-2 content.
- A corrupt/traversing manifest was rejected. A forced rebuild recovered it, scanned retained generation files, and published generation 3 without resetting the semantic generation.
- Manifest entries accept only one local `index-g*.sqlite` file name and reject traversal or missing targets.
- The Wine active database reported schema 15, semantic generation 1, and `PRAGMA integrity_check = ok`.
- The default-path high-frequency incoming query returned a 200-relation page, scanned 8,192 rows, marked the response partial, and completed its service portion in 169.635 ms.
- 661 Rust tests passed; one diagnostic legacy catalog benchmark remained ignored, and 2 LSP smoke tests passed.
- VS Code TypeScript compile/tests passed; architecture fitness reported zero failures and six pre-existing large-file warnings.
