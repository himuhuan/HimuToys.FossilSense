## Scope

This A/B validates removing the `call_strings.text` UNIQUE B-tree from the fact-insert phase of a new full-build database. A build-scoped interner survives all 128-file writer batches, allocates each string ID once, and is released before the explicit `idx_call_strings_text` unique index and the other call indexes are created. Existing and dirty databases keep the online unique index and ordinary `INSERT OR IGNORE` behavior.

The baseline binary is the committed side-by-side/deferred-index build at `62ffb9e`; only call-string uniqueness construction differs.

## Wine A/B

Both sampled runs used the same benchmark runner, release mode, F: benchmark directory, 20 ms process sampling, 7,153 files, 231,403 callable anchors, and 1,714,445 call sites.

A second adjacent unsampled pair showed the same direction: bulk `write_ms` 53.310 s / total 77.555 s versus baseline 68.287 s / total 94.816 s. Absolute times were substantially slower than the earlier deferred-index run on the same machine, so the result establishes the relative optimization, not the <=45 s full-build gate.

The memory increase is build-scoped and matches the 317,390-entry string interner. The map is dropped before secondary index construction and is absent from dirty/query processes. No full-build memory gate is exceeded, but the 51 MiB cost is recorded rather than treated as free.
