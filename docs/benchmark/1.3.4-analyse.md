# FossilSense 1.3.4 大规模索引基准分析

status: current  
version: 1.3.4  
measured_at: 2026-07-12  
binary: `target/release/fossilsense.exe`（`cargo build --release -p fossilsense`）

## 1. 背景

此前以 FFmpeg 作为性能基准，规模约数千文件、数万符号，结构相对规范，难以暴露公司内网目标库（约 **14000 文件 / 66 万符号**、宏与平台分支密集、构建链复杂）中的真实瓶颈。

本次在 `samples/` 引入更“偏激”的公开仓库，并与既有 FFmpeg 对照：

| 样本 | 来源 | 本地路径 | 说明 |
|---|---|---|---|
| U-Boot | `https://github.com/u-boot/u-boot`（shallow） | `samples/u-boot` | 板级/驱动/宏配置密集，接近公司文件与符号规模 |
| Wine | `https://github.com/wine-mirror/wine`（shallow） | `samples/wine` | Win32 API 兼容层，符号密度与调用图远重于文件数 |
| FFmpeg（对照） | 既有 `samples/FFmpeg` | `samples/FFmpeg` | 先前基准，偏小偏理想 |

> Linux 内核曾评估为候选，但 Windows 工作区无法正常检出保留名路径（如 `aux.c`），已放弃；改用 Wine。

样本目录仍受 `.gitignore` 的 `/samples/*` 规则约束（仅 `mini-c` 入库），本地克隆不进入 git。

## 3. 规模对照

| 指标 | 公司目标（参考） | U-Boot | Wine | FFmpeg |
|---|---:|---:|---:|---:|
| C/C++ 扫描文件数 | ~14000 | **13244** | 7153 | 4673 |
| 符号数 | ~660000 | **631893** | 495645 | 85665 |
| callable anchors | — | 91155 | **231403** | 41221 |
| call sites | — | 582841 | **1714445** | 296544 |
| 索引库体积（MB） | — | 313.39 | **587.43** | 105.98 |

结论：

- **U-Boot** 在文件数与符号数上最贴近公司目标（约 0.95× 文件、0.96× 符号）。
- **Wine** 文件数仅约 FFmpeg 的 1.5×，但符号约 **5.8×**，call sites 约 **5.8×**，索引库约 **5.5×**——更适合压调用关系与写放大。
- **FFmpeg** 符号约为 U-Boot 的 **13.6%**，不足以代表 66 万符号量级。

## 4. 全量强制索引（`--force`）

| 阶段 / 指标 | U-Boot | Wine | FFmpeg |
|---|---:|---:|---:|
| elapsed_ms | 52819 | **68441** | 13552 |
| discover_ms | 697 | 278 | 189 |
| check_ms | 7 | 3 | 2 |
| parse_ms | 3271 | **8083** | 2287 |
| write_ms | 47418 | **58761** | 10714 |
| include_edge_ms | 879 | 326 | 184 |
| name_table_ms（CLI） | 0 | 0 | 0 |
| reach_graph_ms（CLI） | 0 | 0 | 0 |
| write / elapsed | **89.8%** | **85.9%** | **79.1%** |
| 符号/秒（粗算） | ~11963 | ~7242 | ~6321 |
| 文件/秒（粗算） | ~250.7 | ~104.5 | ~344.8 |

观察：

1. **写库主导**：三库 `write_ms` 均占 elapsed 约 80–90%。瓶颈首先在 SQLite 持久化与 call/symbol fact 写入，而不是 tree-sitter 解析。
2. **Wine 更慢于 U-Boot**：尽管文件更少，但 call sites 约 3× U-Boot，`parse_ms` 与 `write_ms` 都更高，DB 接近 590 MB。
3. **FFmpeg 给出过于乐观的吞吐**：文件/秒最高，却掩盖了大符号表与大调用图下的写放大。

## 5. 查询侧抽检（冷进程）

每次调用均为新进程：含打开 SQLite、构建 NameTable / RelationCatalog 的成本。

### 5.1 Workspace symbol（`query symbol`）

同一查询连续两次；表内为墙钟 ms。

| 样本 | 查询词 | 命中汇报 | 第 1 次 | 第 2 次 |
|---|---|---|---:|---:|
| U-Boot | `malloc` | 200 / 631893 names | 1689 | 1573 |
| U-Boot | `printf` | 200 / 631893 names | 1571 | 1572 |
| Wine | `CreateWindow` | 200 / 495645 names | 1243 | 1244 |
| Wine | `HeapAlloc` | 47 / 495645 names | 1222 | 1254 |
| FFmpeg | `avcodec_open` | 16 / 85665 names | 308 | 297 |
| FFmpeg | `av_malloc` | 99 / 85665 names | 310 | 302 |

大库 symbol 查询冷启动约 **1.2–1.7 s**，FFmpeg 约 **0.3 s**。进程内重复几乎不降（CLI 无跨请求缓存），说明成本主要在**每次加载 NameTable**，而非单次 ranking。

### 5.2 Call relations（`query calls --incoming`）

| 样本 | 锚点 | relations | catalog_entities | catalog_call_sites | catalog_build_ms | query_us | 墙钟 ms |
|---|---|---:|---:|---:|---:|---:|---:|
| U-Boot | `common/board_f.c` `board_init_f` | 2 | 66738 | 582841 | 2986 | 41 | 3333 |
| Wine | `dlls/ntdll/heap.c` `RtlAllocateHeap` | 159 | 207132 | 1714445 | **8834** | 692 | **9344** |
| Wine（再跑） | 同上 | 159 | 207132 | 1714445 | 8919 | 670 | 9440 |
| FFmpeg | `libavutil/mem.c` `av_malloc` | 513 | 36117 | 296544 | 1399 | 2488 | 1589 |

观察：

- **分页查询本身很快**（数十 µs～数 ms）；墙钟几乎等于 **RelationCatalog 构建**。
- Wine 目录构建约 **9 s**，且 CLI 冷进程无缓存复用；LSP 若每次重建同等代价，打开大库后的 call hierarchy 会明显卡顿。
- U-Boot 虽符号更多，但 call sites 少于 Wine，catalog 构建约 **3 s**，介于 FFmpeg 与 Wine 之间。

## 6. 相对 FFmpeg 的放大倍数

| 指标 | U-Boot / FFmpeg | Wine / FFmpeg |
|---|---:|---:|
| 文件数 | 2.83× | 1.53× |
| 符号数 | **7.38×** | **5.79×** |
| call sites | 1.97× | **5.78×** |
| elapsed_ms | 3.90× | 5.05× |
| write_ms | 4.43× | 5.48× |
| DB MB | 2.96× | **5.54×** |
| symbol 查询（典型） | ~5.2× | ~4.1× |
| catalog_build_ms | 2.13× | **6.31×** |

文件数放大远小于符号/调用图放大；用 FFmpeg 估大库耗时会**严重偏低**。

## 7. 暴露出的真实瓶颈（相对 FFmpeg 基准）

| 优先级 | 瓶颈 | 证据 |
|---|---|---|
| P0 | SQLite 写路径 / fact 落盘 | `write_ms` 占 index 80–90%；Wine 590 MB DB |
| P0 | RelationCatalog 冷构建 | Wine ~9 s / 次进程；查询本身亚毫秒～数毫秒 |
| P1 | NameTable 冷加载 | 大库 symbol 查询稳定在 1.2–1.7 s |
| P2 | 高密度 call sites 的解析成本 | Wine `parse_ms` 3.5× FFmpeg，超过文件数比 |
| — | 发现/校验 | discover/check 可忽略 |

U-Boot 适合回归「接近公司符号规模」的索引写放大；Wine 适合回归「调用关系目录与 DB 体积」极端场景。二者互补，FFmpeg 仅宜作小规模烟雾测试。

## 8. 复现命令

```powershell
$proxy = 'http://127.0.0.1:7720'   # 按本机代理调整
git -c http.proxy=$proxy -c https.proxy=$proxy clone --depth 1 https://github.com/u-boot/u-boot.git samples/u-boot
git -c http.proxy=$proxy -c https.proxy=$proxy clone --depth 1 https://github.com/wine-mirror/wine.git samples/wine

cargo build --release -p fossilsense
$bin = '.\target\release\fossilsense.exe'
New-Item -ItemType Directory -Force -Path target\benchmark | Out-Null

foreach ($pair in @(
  @{n='u-boot'; p='samples\u-boot'},
  @{n='wine';   p='samples\wine'},
  @{n='ffmpeg'; p='samples\FFmpeg'}
)) {
  & $bin index $pair.p --db "target\benchmark\index-$($pair.n).sqlite" --force
}

& $bin query symbol samples\u-boot malloc --db target\benchmark\index-u-boot.sqlite
& $bin query calls samples\wine dlls/ntdll/heap.c 1489 30 --incoming --db target\benchmark\index-wine.sqlite
```

## 9. 小结

| 问题 | 结论 |
|---|---|
| FFmpeg 是否够用？ | 否。符号与调用图量级差一个数量级，会掩盖写库与 catalog 瓶颈。 |
| U-Boot 角色？ | 主规模基准：~1.3 万文件 / ~63 万符号，贴近公司目标。 |
| Wine 角色？ | 调用图/写放大压力基准：更少文件、更多 call sites、更大 DB、更慢 catalog。 |
| 1.3.4 最值得盯的指标？ | `write_ms`、索引 DB 体积、RelationCatalog `catalog_build_ms`、NameTable 冷加载时间。 |
