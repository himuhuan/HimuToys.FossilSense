# FossilSense 1.4.0 large-workspace final gates

status: current

measured_at: 2026-07-12

binary: `fossilsense 1.4.0` release build

databases: schema 15, forced full rebuilds

## Outcome

The large-workspace plan is performance-complete. Schema 15 passes the Wine
writer, call-related storage, and total-database gates. The three-run Wine full
elapsed median is 45.383 s, 0.383 s (0.85%) above the aspirational 45 s gate,
but is 33.7% faster than 1.3.4 and has an explicit phase breakdown. Per the
plan's acceptance rule, no unmeasured follow-up optimization is introduced for
that residual.

Lazy call queries remain bounded well below the 128 MiB Private gate, and the
segmented compact name index passes both the single-file publication and full
cold-build gates.

## Call-query gates

Every row is a fresh CLI process against the final generation. Medians are from
three repetitions.

| Case | Query median | Process median | Peak Private median | Stable bounded result |
|---|---:|---:|---:|---|
| U-Boot board-init incoming | 27 ms | 197.088 ms | 2.27 MiB | 2 returned relations |
| Wine medium fan-in incoming | 26 ms | 215.260 ms | 2.37 MiB | stable relation page |
| Wine high-frequency incoming | 134 ms | 319.259 ms | 14.06 MiB | 8,192 scanned sites, 200 returned relations |

The high-frequency Wine generation produced the same counters on all three
runs of the packaged binary: 1,355 entities, 16,381 compact relations, and
16,382 relation-site refs.
The prior workspace-wide catalog baseline peaked near 1.1 GiB and took 11.8 s;
the final request stays below the plan's 128 MiB Private gate and exposes the
raw-scan/page truncation instead of materializing the tail.

## NameIndex gates

The final Phase 4 A/B used the 495,645-row Wine name table:

| Metric | Result | Gate | Result |
|---|---:|---:|---|
| single-file dirty publication | 10.296 ms median | < 100 ms | pass |
| dirty extra Private | about 11.43 MiB | < 32 MiB | pass |
| full cold NameIndex build | 781 ms median | < 800 ms | pass |
| background compact build | 367 ms median | off publication critical path | pass |

The accepted name-ID postings layout reduced finalization from 276 ms to 130 ms
median. A separate `COUNT(*)` pre-capacity attempt regressed cold publication to
1,133 ms median (+24.9%) and was fully reverted, so it is not part of 1.4.0.
The cold-build gate passed without adding mmap persistence.

The final release audit repeated the same diagnostic after Clippy-only argument
and borrow cleanup: cold build 752 ms, dirty replacement 10.392 ms / 12.08 MiB
Private delta, and background compaction 383 ms. All Phase 4 gates therefore
remain effective in the exact code sent to packaging.

## Correctness and lifecycle gates

- Full and dirty publication do not build a workspace relation catalog.
- Relation requests enforce scan, candidate-expansion, wall-time, call-site,
  and relation budgets; standard LSP adds deterministic caps.
- Per-file overlays shadow base facts without copying the base graph. Save
  cleanup remains content-hash and generation aware.
- Full builds publish side-by-side databases. `IndexDbLease` preserves readers,
  and old generations are deleted only after the last lease is released.
- The final source scan contains no workspace `RelationCatalog`,
  `with_overlays`, `rebuild_relation_catalog`, full call-fact visitor, or
  unlimited `usize::MAX` relation materializer.

## Reproduction

```powershell
cargo build --release -p fossilsense
powershell -NoProfile -ExecutionPolicy Bypass -File scripts/benchmark_large_workspace.ps1 `
  -Repeats 1 -TimeoutSeconds 900 -IncludeFullIndex -CaseFilter u-boot-full-index
powershell -NoProfile -ExecutionPolicy Bypass -File scripts/benchmark_large_workspace.ps1 `
  -Repeats 2 -TimeoutSeconds 900 -IncludeFullIndex -CaseFilter wine-full-index
powershell -NoProfile -ExecutionPolicy Bypass -File scripts/benchmark_large_workspace.ps1 `
  -Repeats 3 -CaseFilter wine-high-frequency-incoming
```
